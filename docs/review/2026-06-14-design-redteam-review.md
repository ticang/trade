# 设计文档红队对抗性复核报告

> 复核对象：`docs/specs/2026-06-14-a-stock-quant-trading-system-design.md`（v0.1）
> 参考资料：`docs/reference-notes.md`
> 复核日期：2026-06-14
> 复核方：独立 Agent（红队 / 对抗性评审，与设计作者隔离）
> 复核原则：找问题而非肯定方案；不确定的技术断言标注"待验证"。

## 总体评价

**结论：需重大修订（不建议作为实现基础直接开工，但架构骨架方向正确，修订后可行）。**

架构分层（数据 / 因子 / 策略 / 风控 / 执行 / 回测 / 可观测）合理，模块化与接口抽象方向对头，"三态同码"作为理想有指导意义。但当前版本有 5 个阻断级问题，任何一个都足以让实现阶段返工；A 股特性清单遗漏会让回测失真；防过拟合机制对系统自身产生的候选爆炸无效；分期路线图无法判定 Done。属"骨架对、血肉缺"状态。

**最关键 3 条修订优先级：**

1. **[Blocker-3] 先做 xtquant 能力探测（M-1 里程碑）**：在未验证 xtquant 实际能力前，⑥ 执行层 / M2 / M4 全部悬空。这是地基假设，必须最先证实或证伪。失败则换执行层。
2. **[Major-1 + Major-2] 重写 A 股特性清单 + 引入 point-in-time 数据语义**：直接决定回测是否可信。漏了最小报价单位 / 收盘集合竞价 / 一字板封单 / 龙虎榜发布延迟的回测，是自欺。M1 前必须完成。
3. **[Blocker-5 + Major-5] 砍掉 ③ 五角色 multi-agent，改为单 agent 起步；路线图验证标准全部量化**：资料原始经验是"手搓了个 multiagents"，非已验证的五角色流水线。先验证"LLM 能否产出有 alpha 的假设"这个核心命题。

**附带强制建议**：M0 之前增加 **M-1 技术可行性验证**里程碑（xtquant + DeepSeek + SQLite/DuckDB 性能三点探测），作为 go/no-go 关口。

> 以上修订已在 v0.2 全部落实。详见设计文档附录 B。

## 一、阻断级问题（Blocker）

### [Blocker-1] 模块编号 ⑧⑨ 在文中被换用，数据流与编号自相矛盾

文档第 2.1 节把"主体行为学习"标 ⑨、"情景模拟与复盘"标 ⑧；但 1.2 目标表（第 31 行）写"情景模拟复盘（⑨）+ 主体行为学习（⑩）"——与全文 ⑧/⑨/⑩ 整体错位一位，会误导据此实现的协作方。

**改进**：统一编号口径为 ⑧ 情景模拟与复盘、⑨ 主体行为学习、⑩ 可观测性，全文核对。

### [Blocker-2] 1.2 表的 ⑩ 指向不同实体

1.2 目标表用 ⑩ 指"主体行为学习"，2.1 表 ⑩ 又是"可观测性"，两处 ⑩ 不是一回事。

**改进**：1.2 表不应复用模块编号指向目标，改用模块名称引用。

### [Blocker-3] xtquant/MiniQMT 同时承担"行情订阅"和"下单"的核心假设未经验证

整套架构把"行情+交易统一接口"压在 xtquant 上，但把它当开箱即用、API 稳定的依赖。现实致命前提：

1. **MiniQMT 开户普遍有 50 万资金门槛**（部分券商上调）。无账户则 ⑥ 执行层 / M2 / M4 全无法验证。
2. **xtquant 行情订阅能力版本差异大**：免费版可能仅日线 + 延迟行情；Level-1 实时 tick 订阅可能需订阅权。4.1 "订阅 tick/分钟线"未说明权限边界。
3. **MiniQMT 进程模型**：xttrader 依赖常驻 mini QMT 客户端进程，IPC 通信，客户端崩溃即下单失败。9 节只笼统写"断线重连"，未处理客户端进程崩溃。
4. **xtquant 是否真支持异步回调式 subscribe(on_bar)** —— 假设了事件回调模型，而 xtquant 回调在内部线程，跨线程给 asyncio 送事件需显式 `loop.call_soon_threadsafe`，文档未提。

**状态**：1–4 均**待验证**（该库无官方文档，靠券商 SDK 内附 PDF/示例）。

**改进**：M0 之前增加 **M-1: xtquant 能力探测**里程碑（最小脚本验证开户可行性、可订阅频率、回调线程模型、断线/客户端崩溃重连、下单延迟）。失败则换执行层选型（easytrader / 券商 OpenAPI）。4.1/4.6 显式声明 xtquant 线程模型，固定一条 thread→asyncio 桥接路径。

### [Blocker-4] "三态同码"在实盘撮合现实性上不成立 —— on_bar 事件驱动 ≠ 实盘低延迟

1. **事件源不同构**：回测 on_bar 由引擎按时间精确派发；实盘 QMT 回调来自 SDK 内部线程，tick/分钟线时序、丢包、延迟不受控。两边 ctx 内容不等价（实盘可能部分成交未回写、持仓未刷新）。
2. **决策延迟**：2.2 表"单标的 < 100ms"，但 on_bar → 因子计算（全市场向量化）→ 策略 → 风控 → 下单串行链路在 Python 单进程下难稳定 < 100ms。无延迟预算分解。
3. **撮合模型不对称**：SimBroker 按 bar 收盘价 + 滑点；实盘 QMT 真实订单簿。一边 bar 收盘成交，一边挂单盘口排队，涨跌停 / 一字板 / 低流动性票行为完全不同。"三态同码"只在接口签名层面成立，**行为层面不成立**。

**改进**：降级表述为"接口三态共用，撮合行为存在偏差，需模拟盘 + 影子模式量化并接受"；4.7 增设"实盘-回测一致性校验"子模块；4.1 实盘额外落 tick / 订单簿快照，否则回测无法复现实盘撮合环境。

### [Blocker-5] DeepSeek "自研轻量 Agent" 用于因子挖掘，严重低估 multi-agent 工程复杂度

1. 资料原文（2.2）只说"手搓了个 multiagents 因子挖掘系统"，无任何工程细节。设计升格为五角色流水线 + 自研调度，等于把**未经验证的口头描述**当已验证架构。
2. 五角色里 Tester 调回测、Judge 做多重校正、Iterator 做变体，每个都是独立工程量；自研调度无重试 / 检查点 / 上下文压缩，跑长链条（100 轮）易崩溃且无法恢复。
3. LLM 输出结构化因子表达式的稳定性，DeepSeek 与 Claude/GPT 差异巨大；Composer 限制为"受限 DSL"但未定义算子集。

**改进**：③ 拆两期——**3a 单 agent 因子假设生成**（LLM 出自然语言假设 + 人/规则翻译），**3b multi-agent 闭环**（3a 验证质量后再上）；DSL 算子集先于 Composer 设计；自研调度状态持久化落 `agent_run` 表。

## 二、重要级问题（Major）

### [Major-1] A 股特性清单（第 8 节）有 6+ 处实质性遗漏

| 遗漏项 | 影响 |
|---|---|
| 除权除息日对策略的影响 | 仅写"前复权"，没区分复权方式对策略扭曲：前复权让历史价被未来除权事件改写，量价类因子（动量/突破）失真 |
| 板块涨跌幅具体值 | 科创/创业 20%、北交所 30%、ST 5%、新股前 5 日不设涨跌幅。撮合必须按板块分支，不能统一 10% |
| 新股次新股 | 上市前 5 日无涨跌停、流动性异常、龙虎榜规则不同。⑨ 行为学习（游资炒次新）和 ① 因子都受影响 |
| 退市整理期 | 30 个交易日价格剧烈波动、流动性枯竭；需 `delisting_period` 标志，策略剔除 |
| 一字涨跌停封单 | 实盘封死一字板根本挂不进单；回测简单按"涨跌停不成交"会高估成交概率 |
| 收盘集合竞价 14:57–15:00 | 14:57 后撤单无效；收盘集合竞价撮合规则；收盘价定义 |
| 最小报价单位 | 随价格档位变化（<0.25→0.001，>1000→0.1）；挂单价须落合法档位 |
| 过户费沪市 | 仅沪市双边征收，2022 年后改双边；撮合需按交易所分支 |
| 融资融券与 T+0 | 是否允许融券 / ETF/可转债 T+0（A 股合法 T+0 通道）未说明 |

**改进**：第 8 节重写为分级表（数据/因子/撮合/风控各列），每条给"实现位置 + 校验规则 + 测试用例"。

### [Major-2] 防过拟合机制对"AI 挖因子 + 行为学习 + 情绪因子"三路并行的候选爆炸严重不足

③ ⑨ ② 三路并行每天产生几十到上百候选因子，而防过拟合武器仅多重假设校正 / walk-forward / look-ahead 检测器。未覆盖：

1. **样本外污染**：walk-forward 样本外段一旦被 Judge 用来选因子即不再样本外。需预留真正不可见 holdout，仅最终验收打开一次。
2. **p-hacking via 变体**：Iterator 参数扰动 / 组合 / 非线性变体每次都是新假设测试，校正次数 N 是否实时累加？不计则一次刷穿显著性。
3. **前视通过情绪 / 龙虎榜 sneak in**：龙虎榜 T 日 18:00 左右公布，若 T 日决策用 T 日龙虎榜 → look-ahead。北向、两融余额、财报均有发布延迟。因子缓存按 (factor, trade_date) 物化，但无 point-in-time 语义。look-ahead 检测器只查代码越界，查不了"数据本身 t 时刻不可得却被当 t 时刻用"。
4. **AI 归纳带训练集偏差**：LLM 读样本归纳，它见过海量历史行情，归纳出的因子可能就是公开知名因子变体，本质过拟合到已知规律。

**改进**：引入 **point-in-time 数据库语义**（每外部源标 available_at，因子计算强制 `available_at <= decision_time`）；预留锁死 holdout 段；Iterator 变体计入 N 并动态调 FDR；AI 归纳增加因子新颖性检验（与已有因子相关 > 0.9 拒绝）。

### [Major-3] SQLite 起步在多策略并发 + factor_value 物化下会成瓶颈，Repository "无成本迁移"是幻觉

1. **factor_value 全市场每日物化**：约 5300 票 × 200 因子 = 106 万行/日写入。WAL 单写者盘后批量可，但回测按 (factor, date_range, universe) range scan，横截面查询 5300 行 × 200 因子 × 回测天数 = 巨量小查询，SQLite 无列存，正是 DuckDB 主场。
2. **多策略并发写**：orders/fills/position/audit/metric 在模拟/实盘被多 asyncio 任务并发写，`database is locked` 常见。9 节没说写锁竞争处理。
3. **Repository 无成本迁移乐观**：SQLite 与 PG/DuckDB 方言（DATE/UPSERT/JSON）、事务、并发模型都不同。Repository 若用 SQLite 特性（WAL/PRAGMA/FTS5），迁移成本非零。接口边界未说，承诺无法证伪。

**改进**：M0 就用 DuckDB 做因子/行情物化（列存，横截面查询快几十倍），SQLite 只放事务表；Repository 接口明确"不暴露任何 SQLite 特性"，M0 即写一份 DuckDBRepository 验证可替换；9 节明确 SQLite 写并发策略（单写队列 + 单写协程串行化）。

### [Major-4] APScheduler + SQLite jobstore + asyncio 在"盘后批量 + 盘中实时 + 跨日恢复"三态下不可靠

1. APScheduler SQLite jobstore **非为高并发或崩溃恢复设计**：崩溃时 job 状态可能停留 RUNNING，重启不自动重跑。
2. **盘前/盘中/盘后流程缺失**：A 股交易日明确节奏（9:15 集合竞价、9:30 开盘、11:30/13:00 午休、15:00 收盘、盘后清算）。节假日调休（A 股特有）会让静态 cron 失效。
3. **APScheduler 与 asyncio 事件循环**：3.x AsyncIOScheduler 在事件循环重启 / 子任务异常时 job 漏执行；4.x API 不稳。

**改进**：调度拆分——盘中事件驱动纯 asyncio（QMT 回调驱动），盘后批处理独立进程（cron/systemd + 交易日历校验），APScheduler 仅轻量定时；新增交易日历驱动调度器（每交易日盘前动态生成 job，处理调休）；job 幂等落 `job_run` 表，重启扫未完成重跑。

### [Major-5] 分期路线图 M2 依赖链断裂，M0→M6 验证标准多数不可量化判定 Done

1. **M2 模拟盘闭环过早**：需 ④ 策略引擎（信号来源）和 ② 因子引擎，但 ④ 无独立里程碑。
2. **风控基础层在 M2 才出现，但 M1 回测已需它**（T+1、涨跌停、100 股取整是回测撮合一部分）。
3. **验证标准不可量化**：M1"结果合理"、M2"跑通"、M3"入库几个？IC 阈值？"、M4"阈值是多少？"——全形容词，无法判 Done。
4. **M5 把 ⑧⑨ 塞一个里程碑**，两者体量巨大，不现实。

**改进**：④ 从 M2 拆出作 M1.5；风控基础层前移 M1；每里程碑验证标准改可判定命题（"夏普>0.5、回撤<30%、look-ahead 0 报警"、"连续 20 交易日对账差异<0.1%"）；M5 拆 M5a / M5b。

### [Major-6] 缺失 5 个关键模块/能力

1. **组合优化器（带约束最优化）**：4.4 组合构建器写"信号→目标持仓+约束"，但整数化（100 股取整）+ 行业暴露 + 换手约束是混合整数规划。无专门优化器会退化为贪心 + 修剪，远离最优。
2. **数据质量校验**：4.1 有清洗但无独立数据质量门禁（缺失率、异常值、停牌填补、复权断裂检测）。脏数据进因子引擎所有下游失真。
3. **参数寻优 / 超参管理**：Iterator 做变体、ML 训练，但无超参管理（网格 / 贝叶斯、版本化、实验追踪如 MLflow）。AI 挖因子是大规模实验，无追踪则不可复现、防不住 p-hacking 统计。
4. **绩效归因系统**：4.7 写"归因（因子贡献/行业贡献）"，但 Brinson / Barra 归因是独立子系统，非几个指标。无归因回答不了"亏钱是因子错还是行业 beta 错"。
5. **灾备与回滚**：9 节有"进程崩溃恢复"但无"数据损坏回滚"（SQLite 文件损坏？因子库被误覆盖？）。

**改进**：新增组合优化器、数据质量门禁，或在现有模块显式补子模块；实验追踪 M3 前就位；盘前盘后流程写进 9 节。

### [Major-7] Agent 因子代码沙箱（白名单算子）与真实因子需求矛盾，且未定义算子集

1. **算子集未定义**：文档从头到尾没列白名单算子。真实因子需 rolling（mean/std/rank/ts_max）、横截面（rank/quantile/zscore）、分组中性化（按行业/市值）、时序对齐（pit）、条件、非线性（sigmoid/tanh）。全在白名单则沙箱接近无限制；不在则产不出有用因子。
2. **白名单与表达力天然矛盾**：业界（Alpha101 / WorldQuant）因子表达式语言允许组合任意算子，纯白名单难表达复杂因子。
3. **沙箱执行 Python 代码真实成本**：即便白名单算子，执行仍需进程隔离（subprocess + seccomp / nsjail）或解释器隔离（RestrictedPython），"沙箱 + 超时 + 资源限额"过于笼统。

**改进**：先定义 DSL 算子集（参考 WorldQuant Brain Operators），M3 前完成；沙箱方案具体化——RestrictedPython + resource.setrlimit + signal.alarm / 子进程。

### [Major-8] 情绪 / 社媒采集合规边界模糊，一期"公开论坛 UGC"仍有 PIPL 风险

1. **公开 ≠ 可自由采集**：股吧/雪球用户发言即便公开也涉个人信息（用户 ID、发言内容）；PIPL 第 13 条对个人信息处理有严格规定，商业用途采集需告知同意。NLP 打分后虽不存原文，但采集过程已构成处理。
2. **平台 ToS**：雪球/东方财富用户协议通常禁止未授权爬取，商业用途可能触发法律风险。
3. **二期社媒爬虫**：抖音/小红书明确反爬，ToS 禁止；合规开关"默认关闭"不等于合规。

**改进**：一期情绪优先用**有授权数据源**（Tushare/AkShare 情绪指数、东方财富开放接口、量化平台现成指标），UGC 改为"仅匿名化聚合指标"（讨论数、关键词词频），不存任何可识别个人内容；合规边界写独立章节，明确采集→存储→使用→销毁全链路 PIPL 清单。

## 三、建议级问题（Minor）

1. **look-ahead 检测器实现未定义**：建议运行时锚（所有数据访问断言 `data.index <= now`），因静态分析覆盖不了外部数据 sneak in。
2. **接口契约表与代码片段命名不完全对齐**：5 节 ScenarioGenerator、4.4 StrategyRunner 等需正向核对。
3. **印花税需注明"以最新政策为准"且支持配置化**（2023.8 从 0.1% 降到 0.05%）。过户费同理。
4. **"中等规模"无量化定义**：建议 1.2 表给资金档位（小资金<50 万，中等 50 万–1000 万）。
5. **NLP 情绪模型选型未定**：金融情绪与通用差异大，建议金融语料微调模型或自标注训练。
6. **可视化与资料"价格+成交量+情绪三合一"对齐不充分**：应在 4.10 显式列为核心交付物。
7. **自身画像（4.9）价值密度低**：单用户小资金自身成交样本极少，统计无意义。建议降级为"交易行为审计 + 规则化提醒"。
8. **测试策略缺 AI 挖因子回归测试**：建议"挖掘确定性测试"——固定随机种子 + 固定 LLM 温度，结果可复现。

## 四、与参考资料对齐度专项

- **multi-agent 因子挖掘**：③ 落地五角色流水线，结构对齐；但资料原仅"手搓了个 multiagents"一句无工程细节，设计扩展为完整架构属过度解读。建议显式标注"③ 工程结构为设计推断，非资料原始经验"。
- **散户情绪反向**：① + ② 对齐；但资料强调"私域群聊采集"，设计推到二期默认关闭——合规必要妥协，但意味着**一期实际放弃了资料的核心差异化 alpha**（公开论坛情绪已被市场充分定价，反向价值有限）。此权衡文档未说透。
- **特殊节点定制策略**：资料强调"针对特殊节点定制策略"，设计仅一句"多策略可插拔 + ⑧ 情景复盘"，无"特殊节点策略工厂"概念。需"快速策略组装 + 短期回测 + 自动下线"流水线。
